# 측량소 — 미래정합성 점수 산식 설계 (Phase 3)

> 작성일: 2026-06-18 · 상태: **제안(검토 대기)** · 승인 전 구현 금지 (CLAUDE.md 하드룰 §4)
> 기준: CLAUDE.md §6 — 미래정합성(0~100) = 테마정합 50% + 재무 30% + 수급 20%

---

## 0. 목적과 하드룰 재확인

측량소는 "내 포트가 **성장 테마**에 얼마나 정렬됐나"를 보여준다. 단일 진실:

- **이건 "성장 렌즈" 하나의 축이다. 보편 점수가 아니다.**
  배당·가치 포트가 낮게 나오는 것은 **결함이 아니라 당연한 결과**다. (하드룰 §4)
- **스타일 중립**: 산식 어디에도 "분산이 좋다 / 배당이 좋다" 같은 우열 가정을 넣지 않는다.
  낮은 점수 = "성장테마 비중이 낮다"는 **중립적 사실**일 뿐.
- **종목별 기여도를 투명하게** 보여준다 (점수가 어디서 왔는지 분해).
- 모든 응답에 **"투자권유 아님"** disclaimer (이미 `common.DISCLAIMER`로 적용 중).
- 매수/매도 조언 생성 금지. 점수는 "현재 상태 측정"이지 "이걸 사라/팔라"가 아니다.

## 1. 데이터 확보 가능성 점검 (먼저 — 이게 게이트)

Phase 2와 동일하게 "되는지"부터 확인. 웹 조사로 엔드포인트 존재 확인 완료, **실계좌 재확인은 구현 직전 게이트**(아래 §10 체크리스트).

| 구성요소 | 필요 데이터 | 소스 | 가능? | 핵심 리스크 |
|---|---|---|---|---|
| **테마정합 50%** | 테마 ↔ 종목 매핑 | 키움 `ka90001`(테마그룹별), `ka90002`(테마구성종목) | ✅ | 키움 테마는 **키움이 큐레이션**한 목록 → "성장테마"는 우리가 화이트리스트로 별도 정의해야 함. rate limit(~1req/s) → 캐시 필수 |
| (보조) | 업종(섹터) 분류 | 키움 `ka10001`(주식기본정보, 업종 포함), `ka10101`(업종코드) | ✅ | 섹터는 거칠다(AI·로봇·SMR 같은 교차테마 못 잡음) → 폴백/검증용 |
| **재무 30%** | 매출·영업이익(성장률) | **DART** `fnlttSinglAcntAll` | ✅ | ① DART 무료 API키 발급 필요 ② ticker→`corp_code` 매핑 테이블 필요 ③ 분기 공시 지연(최대 3개월) ④ ETF·리츠는 단일회사 재무 없음 → 제외 처리 |
| **수급 20%** | 외인·기관 순매수 | 키움 `ka10059`(종목별투자자기관별), `ka10008`(외국인 종목별 매매동향), `ka10009`(기관) | ✅ | 데이터 지연(§8 기지정), 모멘텀성 노이즈 → 가중치 20%로 제한 |

**결론: 세 구성요소 모두 확보 가능.** 단, 구현 전 선결 2가지:

1. **DART API 키 발급** (무료, 즉시 — opendart.fss.or.kr 가입 → 키 발급) → `.env`에 `DART_API_KEY=` 추가
2. **corp_code 매핑** — DART `corpCode.xml`(전종목 corp_code 덤프) 1회 다운로드 → 로컬 테이블화 (ticker↔corp_code). 갱신은 월 1회면 충분.

> 키움/DART 키·시크릿은 모두 `.env`, 서버 보관, git·프론트 노출 금지 (하드룰 §4).

## 2. 전체 산식 구조 (총점 + 기여도 정확 합산)

각 종목 `i`마다 0~100 서브점수 3개를 구하고, 포트 비중으로 가중합한다.

```
포트비중 wᵢ   = 종목 i 평가금액 / 전체 평가금액      (Σwᵢ = 1)
종목점수 sᵢ   = 0.5·theme_i + 0.3·fin_i + 0.2·supply_i   (0~100)

미래정합성  S = Σᵢ ( wᵢ · sᵢ )                          (0~100)
기여도   cᵢ  = wᵢ · sᵢ                                  (Σcᵢ = S, 정확히 합산)
```

- 가중치 합 = 100 (테마50/재무30/수급20) — CLAUDE.md §6 그대로.
- **기여도 `cᵢ`는 총점 `S`로 정확히 합산된다** → "이 88점은 어느 종목에서 왔나"를 수학적으로 정확히 분해 가능 (투명성 하드룰 충족).
- 현금·ETF·리츠 등 재무/테마 산정 불가 종목은 §6 규칙으로 처리.

### 워크드 예시 (3종목)

| 종목 | 비중 wᵢ | theme | fin | supply | sᵢ = 0.5T+0.3F+0.2S | 기여 cᵢ |
|---|---|---|---|---|---|---|
| SK하이닉스 | 35% | 100 (AI·반도체✓) | 85 | 80 | 91.5 | 32.0 |
| 삼성전자 | 30% | 100 (반도체✓) | 70 | 60 | 83.0 | 24.9 |
| KB금융 | 35% | 0 (성장테마 미매칭) | 75 | 50 | 32.5 | 11.4 |
| **합계** | 100% | | | | | **S=68.3** |

→ KB금융은 **theme=0이라 점수를 끌어내리지만, 이는 "나쁜 종목"이 아니라 "성장테마에 안 속함"이라는 중립 사실.**
재무·수급으로는 정상 기여(11.4점). 이 예시를 UI 설명 문구의 근거로 사용.

## 3. 테마정합 (50%) — 상세

### 정의
"포트폴리오가 **우리가 지정한 성장테마**에 얼마나 들어가 있나."

### 데이터 흐름
1. **성장테마 화이트리스트**(우리가 큐레이션, 버전 관리) — 예: `AI·반도체`, `HBM`, `전력기기·전력망`, `로봇`, `2차전지`, `바이오`, `SMR·원전`, `우주·방산`, `자율주행`.
   - `backend/app/data/growth_themes.py` 또는 `config/growth_themes.json`으로 분리 → **사용자에게 전체 목록 + 선정 이유를 그대로 노출**(렌즈의 정의 그 자체).
2. 키움 `ka90001`로 전체 테마그룹 → `ka90002`로 각 테마 구성종목 → **역인덱스**(종목→속한 테마 집합) 구축, **캐시**(일 1회 갱신, rate limit 회피).
3. 화이트리스트와 교집합 → 종목별 "매칭된 성장테마" 도출.

### 종목 서브점수 `theme_i`
- **v1 (단순·방어적)**: 매칭되면 100, 안 되면 0.
  → 포트 가중합 = **"성장테마에 속한 평가금액 비중(%)"** = 직관적이고 조작 어렵고 스타일 중립적(구조적으로).
- **v1.1 옵션(강도 반영)**: `theme_i = min(100, 60 + 20×(매칭 테마 수))` — 여러 핫테마 교차 종목 가산. (단순함 우선이면 보류)

### 리스크/주의
- 키움 테마는 키움이 정의 → 우리 화이트리스트로 "어떤 테마가 성장인가"를 **명시적·공개적**으로 선택. 이게 곧 "성장 렌즈"의 정의.
- 테마 미매칭 = 0점이지 **음수 아님**. 부재는 감점이 아니라 "기여 없음".

## 4. 재무 (30%) — 상세

### 정의
**성장 관점**의 재무 체력. 성장 렌즈이므로 **성장률 중심**(매출·영업이익 성장), 가치지표(저PER·저PBR)나 배당수익률은 쓰지 않는다. (배당·가치 우대 금지 — 하드룰 §4)

### 데이터 (DART `fnlttSinglAcntAll`)
종목별 corp_code로 최근 연간(또는 직전 4분기 누적) 조회:
- 매출액, 영업이익, 당기순이익 → 전년 대비 성장률(YoY). 가능하면 3년 추세(안정성).

### 종목 서브점수 `fin_i` (제안)
```
fin_i = 0.4·norm(매출성장률) + 0.4·norm(영업이익성장률) + 0.2·norm(이익률/흑자지속)
```
- `norm()`: 성장률을 0~100으로 매핑하는 **상하한 곡선**(예: 0%→50, +25%↑→90+, 역성장→하한 floor지만 0은 아님).
- 적자→흑자전환, 흑자지속 등은 마지막 항으로 가산.

### 리스크/주의
- **분기 공시 지연**(최대 3개월) → 응답에 "재무 기준일" 표기.
- **ETF·리츠·스팩**: 단일회사 재무 없음 → `fin_i` 산정 불가 → §6 규칙(중립 처리 + 플래그).
- 이건 **성장 재무 렌즈** — 안정적 고배당·저성장 우량주가 여기서 낮게 나오는 건 **설계상 정상**이며 "나쁜 회사" 신호가 아님(문구로 명시).

## 5. 수급 (20%) — 상세

### 정의
최근 외국인·기관의 순매수 흐름(스마트머니 시그널). 방향성 지표라 스타일 중립.

### 데이터 (키움 `ka10059` 우선, `ka10008`/`ka10009` 보조)
- 종목별 최근 N거래일(제안: **20일**) 외인+기관 누적 순매수.

### 종목 서브점수 `supply_i` (제안)
- 순매수 금액을 거래대금(또는 시총) 대비 정규화 → 0~100 버킷.
- 순매도면 낮게(하한 floor), 순매수면 높게. 단 가중치 20%라 총점 영향 제한적.

### 리스크/주의
- **수급 지연**(§8) → "수급 기준일" 표기.
- 모멘텀성·노이즈 → 윈도우를 너무 짧게 잡지 않음(20일 제안).

## 6. 산정 불가 종목 처리 (현금·ETF·리츠 등)

스타일 중립·투명성 위해 **일관 규칙** 필요:

| 종목 유형 | theme | fin | supply | 비중 처리 |
|---|---|---|---|---|
| 일반 주식 | 정상 | 정상 | 정상 | 정상 |
| ETF/리츠/스팩 | 테마 매칭 가능시 정상, 아니면 0 | **N/A → 해당 항 제외, 가중치 재정규화** | 가능시 정상 | 비중 유지 |
| 현금 | 0 | N/A | N/A | **분모(전체)에는 포함하되 점수 0 기여, UI에 "현금 비중"으로 별도 표기** |

- 재무 N/A인 종목은 그 종목 내에서 `0.5/0.2`만으로 재정규화(테마·수급만). **0점으로 깎지 않음**(데이터 부재 ≠ 결함).
- 모든 N/A는 응답에 `flags: ["fin_na"]` 식으로 **투명 표기**.

## 8. 스타일 중립성 보장 장치 (하드룰 검증)

- 산식에 분산·종목수·배당 가점 **없음**. 비중 가중합만.
- 미매칭/N/A는 **0 기여**일 뿐 **음수 감점 없음**.
- UI 색: 낮은 점수를 빨강(=나쁨)으로 칠하지 않음. **중립 톤** + "성장테마 비중이 낮습니다(결함 아님)" 문구.
- 등급(A~D)은 "성장테마 정렬 강도" 라벨이지 "포트 우열"이 아님을 명시.

## 10. 구현 직전 실계좌 재확인 체크리스트 (게이트)
