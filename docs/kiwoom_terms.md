# 키움 REST API 약관 조사 — 멀티유저 SaaS 가능 여부 게이트

> 조사일: 2026-06-25
> 핵심 질문: "제3자 서버에 사용자가 자신의 앱키·시크릿을 입력해 사용"이 약관상 허용인가?

---

## 판정: **명시적 금지에 가까운 회색지대**

---

## 1. 키움 REST API 자체 약관

공개 문서에서 "상업적 이용 금지" · "앱키 양도 금지"를 명문으로 규정한 조항은 확인되지 않음.
그러나 서비스 구조 자체가 개인 사용을 전제로 설계되어 있음.

**구조적 금지 요소 (openapi.kiwoom.com 서비스 안내):**

| 항목 | 내용 |
|---|---|
| IP 등록 제한 | **허용 IP 최대 10개**만 등록 가능 |
| 앱키 발급 | **1회만** 가능 (재발급 없음) |
| 계좌 보유 조건 | "당사 계좌를 보유한 고객에게만 제공" |

멀티유저 SaaS와의 충돌 포인트: 각 사용자의 앱키마다 우리 서버 IP를 등록해야 함.
기술적으로 가능하지만, 키움이 의도한 "내 PC에서 내 API" 사용 방식과 완전히 다른 구조.

---

## 2. 업계 표준 약관 (한국투자증권·토스증권에서 확인)

**토스증권 오픈 API 이용약관 (명문 확인):**
> "고객은 시세 정보를 **개인투자자 본인의 매매 목적에 한하여** 이용하여야 하며,
> **제3자에게 제공하거나 배포 또는 상업적으로 활용해서는 아니 된다.**"
> "법인 고객은 오픈 API 서비스 대상에서 제외"

이것이 한국 증권사 업계 표준. 키움에도 실질적으로 동일한 조항이 있을 가능성 매우 높음.

**한국투자증권 오픈 API 약관 (구조 확인):**
제3자 서비스(SaaS)는 허용하되, 반드시 증권사와 별도 **"이용기관 계약"** 체결 필요.
개인이 임의로 앱키를 제3자 서버에 입력하는 방식이 아니라 B2B 계약 구조.

---

## 3. 핵심 질문에 대한 판정

| 판단 기준 | 판정 |
|---|---|
| 키움 약관 명문 조항 | 명시적 금지 조항 미확인 (공개 문서 기준) |
| 업계 표준(토스·한투) | **명시적 금지**: 개인 투자 목적 한정, 제3자 제공·상업적 활용 금지 |
| 키움 REST API 구조 | IP 등록·앱키 1회 발급 등 개인 사용 전제 설계 |
| 한국투자증권 제휴 구조 | 제3자 서비스는 이용기관 계약 필수 |
| **종합 위험도** | **높음** |

---

## 4. 실무적 위험 시나리오

- 키움이 약관 위반으로 앱키 **강제 만료** → 서비스 중단
- 키움이 **계좌 이용 제한** 조치
- 금융당국이 "무인가 투자일임"에 준하는 것으로 해석할 여지 (계좌 대행 성격)
- 사용자 앱키·시크릿 서버 저장 → **전자금융거래법상 접근매체 보관 문제** 가능성

---

## 5. 합법적 대안

| 방안 | 내용 | 시점 |
|---|---|---|
| **이용기관 계약** | 증권사와 B2B "제휴 이용기관" 계약 → 사용자 위임 구조 합법화 | v2, 자본금·심사 필요 |
| **마이데이터 허가** | 금융위원회 마이데이터 사업자 허가 (자본금 5억+) | v3, 투자 유치 후 |
| **로컬 실행 구조** | 앱키가 사용자 기기에만 있고 서버는 조회만 중계 (앱키 서버 비저장) | 기술 복잡도 높음 |
| **싱글유저 유지 (현재)** | 본인 계좌·앱키·서버 → "개인 투자 목적" 범주 | v1.5 현재 구조 |

---

## 결론

**v1.5 (Railway, 본인 혼자):** 위험 낮음. 현재 구조 그대로 진행 가능.

**v2 멀티유저 전환 전 필수:**
키움증권에 "제휴 이용기관 API 계약 가능 여부" 공문 문의 선행.
한국투자증권은 이미 이 구조를 허용 중이므로 키움도 협의 가능성 있음.
무단 멀티유저 운영 시 계정 차단·서비스 중단·법적 리스크.

**v2 위험 완화 설계 옵션:**
앱키를 서버에 영구 저장하지 않고 사용자 기기에서 암호화 전송 후 즉시 폐기하는 구조.
전자금융거래법상 "접근매체 보관" 리스크 경감 가능.
