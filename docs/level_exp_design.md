# 레벨 / EXP 시스템 설계 (level_exp_design.md)

> 상태: **설계안 (검토 대기)**. 코드 작성 전. 승인 후 구현.
> 관련: CLAUDE.md §4 하드룰, §5 HUD, §6 티어/XP. 이 문서가 티어 시스템을 대체한다.

## 0. 배경 & 목표

**무엇을 바꾸나**
- 기존 **티어(아이언~챌린저 / 디비전 IV~I)** 제거. 이유: 측량소 등급(`광맥 A·88`)과 의미가 중복되고, "상위 N%" 표기가 종목 추천/우열 평가처럼 읽힐 위험.
- 대신 **레벨 1개로 통일** (`Lv 7` 식 표기) + 그 아래 **EXP 게이지**.
- 실제 **익절·배당**이 들어오면 게이지가 차오르고 레벨업. **"배당 들어오니 레벨업하네?"** 의 즐거움이 핵심 동기.

**무엇을 유지하나**
- 측량소 **펀더멘털 등급(`광맥 A·88`)** 은 그대로. 레벨은 티어만 대체. HUD 2단의 등급칩·성향칩은 유지.

**현재 코드 영향(요약)**
- `scoring_service.tier()` 는 현재 `{"name":"마스터","division":4,"emblem":"master"}` **하드코딩 스텁** → 실제 로직 없음. 제거 부담 작음.
- HUD 1단(`Hud.jsx`)의 티어 엠블럼/이름/`광부 등급·상위 N%`(`TIER_RANK`) → 레벨 배지 + EXP 게이지로 교체.

## 1. ★ 하드룰 충돌 해결 (가장 중요)

### 1.1 무엇이 충돌하나
CLAUDE.md §4: **"XP는 원(₩) 액수에 비례 금지. 손실로 티어 강등 금지."**

→ "100원 = 1 EXP" 같은 **절대 금액 방식 절대 금지.**
이유(앱 철학 위반):
- 종잣돈 큰 사람이 자동으로 고레벨 → 작게 시작한 사람 영구 좌절.
- "레벨 올리려면 더 넣어"라는 **도박적 입금 압박** 유발.
- 광맥의 철학("얼마를 넣었나가 아니라 얼마나 건강하게 굴렸나")과 정면 충돌.

### 1.2 해결 원칙 (4개)
1. **금액(₩)이 아니라 비율(%)·행동·연속성으로 EXP 부여.** 종잣돈 크기와 무관.
2. **EXP는 단조 증가만.** 손실은 EXP 0 (감점 없음). **레벨은 절대 내려가지 않는다.** (손실 무강등 충족)
3. **스타일 중립.** 특정 투자철학을 우대하지 않는다 → 모든 성향이 레벨업 경로를 가진다 (§1.3).
4. **사이즈 무관 검증.** "종잣돈 1천만 vs 1억"이 같은 수익률·같은 행동이면 **같은 EXP** 가 나와야 한다. (설계 검수 기준)

### 1.3 스타일 중립을 어떻게 보장하나 (핵심 논증)
배당만 보상하면 "배당주 우대"가 되어 §4 위반. 그래서 **3개 EXP 소스가 각 성향의 길을 동등하게 연다**:

| EXP 소스 | 주로 보상받는 성향 | ₩비례? |
|---|---|---|
| ① 배당 수확 | 배당·인컴형 | ✕ (이벤트당 고정) |
| ② 익절(실현 수익률 %) | 성장·매매 실현형 | ✕ (% 기반) |
| ③ 행동·연속성·장기보유 | 매수후보유·장기 규율형 | ✕ (행동 기반) |

- **장기 보유자**(거의 안 팖, 배당 적음)도 ③으로 레벨업 → 굶지 않음.
- **성장 실현형**(배당 안 받음)도 ②로 레벨업.
- **배당형**도 ①로 레벨업.
- → 어떤 스타일이든 "꾸준히·잘 굴리면" 오른다. 한 철학도 우대 안 함.

> UI 고정 문구(레벨/측량소 톤 유지): **"레벨은 투자 실력·우열이 아니라 '꾸준히 굴린 기록'입니다. 투자권유 아님."**

## 2. EXP 환산 공식

> 표기: `EXP`는 정수. 모든 상수는 **튜닝 가능**(아래는 v1 제안값). 금액(₩) 입력은 어떤 공식에도 들어가지 않는다.

### 2.1 ① 배당 수확 EXP
```
E_div(이벤트 1건) = DIV_BASE                       (= 100, 금액 무관 고정)
  (+ 선택) 배당수익률 보너스 = round(BONUS_K · min(yield%, YIELD_CAP))
           yield% = 배당금 / 해당 종목 보유원가          ← Phase 2, 기본 OFF
```
- **Phase 1: 배당 1건당 무조건 +100.** 1만원 배당이든 100만원 배당이든 동일. (₩완전 무관, 가장 깔끔)
- yield 보너스는 원가(cost basis) 신뢰도 확보 후 Phase 2에서. yield는 **비율**이라 ₩비례 아님(허용). 단 매도분 배당은 원가 추적이 어려워 보류.
- 안티-파밍: 배당은 사용자가 임의 생성 불가(실제 입금/CSV 기록) → 파밍 위험 낮음.

### 2.2 ② 익절(실현 수익률) EXP
데이터: `ka10073` 의 `returnRate`(rlzt_rt, 실현수익률 %). **`realizedPnl`(₩)는 사용 안 함.**
```
손실(realizedPnl ≤ 0)        → E_sell = 0          (감점 없음, 단조 증가 유지)
이익(returnRate% = r > 0)    → E_sell = round( SELL_W · clamp(r, 0, SELL_CAP) )
                               SELL_W = 10,  SELL_CAP = 30(%)
```
- 예: +10% 익절 = 100 EXP. +30% = 300 EXP. **+80% 도 300 EXP(상한)** → 한 방 대박이 레벨을 지배하지 못함.
- **하한 컷**: r < MIN_R(=1%) 이면 0 EXP → 0.1% 초단타 스캘핑 파밍 차단.
- **일일 상한**: 익절 EXP는 하루 합산 DAILY_SELL_CAP(=600) 까지 → 잘게 쪼개 팔아 파밍 차단.
- (대안 곡선: 제곱근형 `SELL_W·√r` 도 검토했으나, **선형+상한**이 사용자에게 설명하기 가장 투명("1% = 10 EXP, 한 번 최대 300") → 추천.)

### 2.3 ③ 행동·연속성·장기보유 EXP
| 행동 | EXP | 메모 |
|---|---|---|
| 일일 앱 열람(연속) | +STREAK_BASE(=10) ·연속일 가중, 주간 마일스톤 +50 | 신규 추적 필요 |
| 장기보유(월 1회 정산) | 30일+ 보유 종목 1개당 +HOLD(=20) | 보유 스냅샷 비교 |
| 기록 완성(거래/배당 임포트) | 건당 +REC(=5), 1회성 | 기록 노가다 제거 가치 강화 |
- 연속성/장기보유는 **금액·수익률과 완전 무관** → 가장 순수한 "규율" EXP.
- ③은 신규 추적 인프라 필요(§6.3).

### 2.4 멱등성 (가장 중요한 구현 제약)
EXP는 **이벤트 원장(ledger)** 에서 합산한다. 같은 익절/배당을 재조회해도 **중복 적립 금지**, EXP는 **오직 증가만**.
```
exp_events(ref_key UNIQUE) → 신규 이벤트만 INSERT → total_exp = Σ exp
```
`ref_key` 예: 익절=`SELL|date|ticker|qty|sellPrice`, 배당=`DIV|date|ticker|amount`. (상세 §6.4)

## 3. 레벨업 곡선

### 3.1 공식 (다항식 — 권장)
```
need(L) = 다음 레벨(L→L+1)에 필요한 EXP = round(BASE · L^EXP_P)
BASE = 100,  EXP_P = 1.5
cum(L) = Σ_{k=1..L-1} need(k)   (레벨 L 도달까지 누적)
```
- **지수형(`BASE·g^L`) 은 비추천**: 후반부 숫자가 비현실적으로 폭발 → 동기 상실.
- 다항(L^1.5)은 "초반 빠른 성취감 + 후반 완만한 도전"의 균형.

### 3.2 곡선 표 (BASE=100, P=1.5)
| 구간 | need(L) | 도달 누적 cum |
|---|---|---|
| Lv 1 → 2 | 100 | 0 |
| Lv 2 → 3 | 283 | 100 |
| Lv 5 → 6 | 1,118 | ~1,400 |
| Lv 10 → 11 | 3,162 | ~12,000 |
| Lv 30 → 31 | 16,432 | ~약 23만 |
| Lv 50 → 51 | 35,355 | ~약 96만 |
| Lv 99 → (cap) | 98,505 | ~약 590만 |

해석(체감): +10% 익절 1번 = 100 EXP. **Lv1→2는 익절/배당 1~2번**으로 즉시 → 초반 "오 오르네!" 확보. 후반은 꾸준함 누적이 필요.

### 3.3 튜닝 레버
`BASE`(초반 난이도) · `EXP_P`(후반 가팔라짐) 두 상수로 전체 페이스 조절. 실데이터(실현손익·배당 빈도) 관측 후 조정.

## 4. 만렙 — Lv 99 vs 무한

| | A. **유한 (Lv 99 cap)** | B. 무한 |
|---|---|---|
| 장점 | 명확한 엔드게임("99" 자체가 게임적 상징), 도달 시 성취·명예 | 평생 목표 유지, 장기/헤비 유저 이탈 방지 |
| 단점 | cap 도달자 진행감 소멸 → 재참여 동기 약화 | 숫자 무한 팽창 → 의미 희석, "명예의 순간" 없음 |

### ★ 추천: **유한 Lv 99 (v1)** + v2 프레스티지 훅
- v1은 **Lv 99 상한**. 곡선상 590만 EXP ≈ 충분히 장기 목표(수년). 대부분 유저는 cap 근처에 못 가므로 v1에서 cap 소멸 문제는 비현실.
- cap 도달자가 의미 있게 생기면(리텐션 데이터 확인 후) **v2에서 "광맥 깊이/명성(프레스티지)" 루프**: Lv 99 → "환생"하여 깊이 단계↑ + 배지. v1에서는 **설계만 열어두고 미구현.**
- 결론: **숫자 상한 99로 고정하되, 데이터 모델은 무한 EXP를 저장**(total_exp는 그냥 누적, 레벨만 99에서 클램프) → 나중에 프레스티지로 확장 자유.

## 5. EXP 데이터 소스 확보 가능성

### 5.1 ② 익절 — ka10073 (✅ 확보됨)
- `kiwoom_client.get_realized_pnl()` 이 이미 `returnRate`(rlzt_rt) 반환 → **% 기반 EXP 즉시 가능.**
- 주의: 현재 조회 창이 최근 365일. **신규 이벤트만** 적립하려면 원장으로 중복 제거 필수(§6.4).

### 5.2 ① 배당 — CSV 임포트 (✅, 단 게이트 있음)
- 키움 배당 API 없음 → **CSV 임포트 기록(DB)만**. `Dividend` 테이블 각 행 = 1 이벤트.
- 즉 배당 EXP는 **사용자가 CSV를 넣어야** 적립 → 온보딩에서 임포트 유도 필요. (기능 자체는 확보됨)

### 5.3 ③ 행동·연속성 — (⚠️ 신규 인프라)
- 앱 열람 로그 / 장기보유 스냅샷 / 기록 이벤트 = **현재 추적 안 됨.** 신규 테이블·잡 필요.
- → 단계적: Phase 1은 ①②만, ③은 Phase 2.

### 5.4 멱등 원장 (신규, 공통 필수)
- 신규 테이블 `exp_events(id, scope, source, ref_key UNIQUE, exp, created_at, meta)`.
- 적립 잡: vault 갱신 시 realized_pnl·dividends 훑어 **미적립 ref_key만 INSERT**.
- `total_exp = SUM(exp)` → 레벨/게이지 파생. (단조 증가·중복 방지 자동 충족)

## 6. 데이터 모델 / 영향 범위

### 6.1 제거
- `Hud.jsx`: `TIER_RANK`, 티어 엠블럼/이름/`광부 등급·상위 N%` 블록(1단).
- `scoring_service.tier()` 스텁 + 호출부(`portfolio_service` line 89 `"tier": ...`).
- 9종 티어 엠블럼(`assets/emblems/*`)은 티어용으로는 미사용 → **레벨 배지 소재로 재활용 검토**(§7).

### 6.2 신규
- 테이블 `exp_events` (멱등 원장, §5.4).
- (Phase 2) 테이블 `activity_log` / `holding_snapshot` (③ 행동 EXP용).
- API: portfolio 응답에 `level` 객체 추가 (§6.3).

### 6.3 응답 스키마(제안)
```
"level": {
  "level": 7,
  "totalExp": 12840,
  "curLevelExp": 840,        // 이번 레벨 진행분
  "needExp": 3162,           // 이번 레벨 요구량 need(7)
  "ratio": 0.27,             // 게이지 채움 비율
  "isMax": false
}
```
`tier` 키는 제거(프론트 동시 수정).

### 6.4 멱등 키 규칙
- 익절: `SELL|{date}|{ticker}|{sellQty}|{sellPrice}`
- 배당: `DIV|{date}|{ticker}|{amount}`
- 행동: `STREAK|{date}` / `HOLD|{yyyymm}|{ticker}` / `REC|{trade_id}`

## 7. 레벨 / 게이지 UI 컨셉 방향

> 상세 디자인은 구현 단계. 여기서는 **방향만.**

### 7.1 위치·크기 (게이지를 홈의 최고 재미 요소로)
- HUD **1단을 레벨 전용 띠로 교체**: 좌측 **레벨 배지(크게)** + 우측 아이콘(🔔⚙) 유지.
- **EXP 게이지 = 1단 바로 아래 풀폭 가로 바**(눈에 확 띄게). 금색이 차오르는 "광맥 채굴 게이지".
- 게이지 위/안에 마이크로 라벨: `840 / 3,162 EXP · 다음 레벨까지 2,322`.
- 2단(등급칩·시장토글·성향칩), 3단(총 평가금액)은 유지.

### 7.2 레벨 숫자 디자인 ("오 뭐지 재밌네?")
- 표기는 **`Lv 7`** 고정. 하지만 **금광/광맥 컨셉의 결정(crystal) 배지** 안에 박힌 형태.
  - 금색 그라데이션 숫자 + 내부 글로우(따뜻한 빛, §5 L2 오로라와 톤 통일).
  - 육각 메탈 + 수정(기존 엠블럼 모티프 재활용 가능) → **레벨이 오를수록 결정 면(facet)·글로우 강도 증가** = 시각적 성장.
  - 컨셉 라인: 레벨 = **"광맥의 깊이/순도"**. "Lv 7 = 7번째 갱도" 같은 광부 서사.

### 7.3 레벨업 순간 연출
1. 게이지가 100% 도달 → 금색 가득 + 가장자리 shimmer.
2. **터지는 이펙트**: 결정 배지가 쩍 갈라지듯 빛 폭발 + 금가루 파티클(기존 L3 파티클 레이어 재활용).
3. 숫자 플립 `7 → 8` + bloom.
4. `Lv 8!` 토스트 + (옵션) 햅틱/사운드.
5. **금광 메타포 연계(권장)**: 수레(cart)가 게이지로 금을 쏟아붓는 트윈 → 가득 참 → 레벨업. "배당 수레가 레벨을 올린다"는 인과를 시각화.

## 8. 단계적 구현 제안

| Phase | 범위 |
|---|---|
| **P1** | 멱등 원장 `exp_events` + ② 익절(%) + ① 배당(이벤트당 고정) EXP + 레벨 곡선 + `level` 응답 + HUD 레벨 배지·게이지(정적) |
| **P2** | ③ 행동·연속성·장기보유 EXP(추적 인프라) + 레벨업 연출(파티클/수레 트윈) + 배당 yield 보너스 |
| **P3(v2)** | 프레스티지("광맥 깊이") 루프, 성향 색 스킨 연계 등 |
